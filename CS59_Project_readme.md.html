P00 Project Selection and Approval ================= Define the domain. ---------------------------------- The domain for our language is education, specifically lower-school teaching. Define the problem. What is the purpose of the language? ---------------------------------- Teachers in lower school may not have time to effectively generate exams/quizzes for their students. Additionally, quizzes are often homogenous across a classroom, and can lead to cheating. The purpose of our language is to provide an easy, intuitive solution for these teachers. Our language will allow the creation of quizzes for arithmetic problems. The generator will feature a question bank which the generator will randomly pull from to generate multiple choice questions. Additionally, the numbers present in the problems will be randomly generated based on a range given in the input, to ensure students canâ€™t look at other tests. Example solutions and outputs. ---------------------------------- In this process the teacher would write in expressions such as ax + b = y. Our DSL would randomly generate values for the constants a and b. As well as a randomly generated y value for the solution. Example Problem: ax + b = c, What is the value of x? An expected output would be: 6x + 11 = 47. Randomly generated answers for the problem would be a. 4, b. 9, c. 6, d. 2, with 6 being the correct answer. The quiz taker would then solve for the value of x. Defining functionality. ---------------------------------- Generally speaking, and subject to change, a teacher provides a list of questions in the correct format that would fall into the question bank. The teacher would also provide a question count and constant range, which would be used to create different numbers in the question. For any given question, 4 potential answers are given. Three of the multiple choice answers are randomly generated ints, while the last is determined based on the question itself and the randomly generated constants, or in other words, the actual answer. All of these questions are compiled together in an html-like file in a readable format. Approval? ---------------------------------- Approved over slack. Our implementation assumes a standard, four answer multiple choice format for the quiz. As of now, the grammar and translation is only able to handle input from linear equations. Any given question will have 4 possible answers, listed A through D, and one of them will randomly be assigned the actual answer for the equation. Questions from the input bank are selected at random, and the question count of the quiz is based on the "Count: #" input. Every question is evaluated to determine its correct answer, storing the results in an answer bank that is printed at the bottom of every output, after the markdeep formatting footer. In order to solve the equations to determine the correct answer for a given question, the Symja Android Math Library was used. In particular, the Roots() function was used to solve for the x values in each equation. In the future, if different types of questions were desired in the quiz, Symja could also be used to solve for a system of equations, as well as more complex polynomials. The link to the Symja github page is [in this hyperlink](https://github.com/axkr/symja_android_library). See the solveEquation() function in our implementation to see how it was used. Every capital letter in a given question from the bank has a random number assigned to it from within the range designated for the question. The range allows the user to decide the size of the numbers in the question, as working with smaller numbers can sometimes be easier. For example, if the given range of a question is (3 to 50), the only numbers that could replace the capital letter constants in an equation such as -Ax = Bx - C are between 3 and 50, inclusive. An example of the variable assignment being completed for this example may look like this: -8x = 12x - 37. The solveEquation() function would then solve this equation for x, note the answer, and randomly include it as one of the 4 potential answers to the question, as well as keep track of it for the answer bank. The final part of the listener that is randomized is the other potential answer choices for each questions. As it stands right now, the potential answers are chosen based on a range twice as big as the range for the specified question, but ultimately shifted in the negative direction by the "range". If the given range is (1 to 100), the possible answer choices that could show up for that question can range from -100 to 100, inclusive. This allows for negative answers to be a possibility, and provides an estimate of where the actual answer may fall. Ultimately, the translation to Markdeep formatting wasn't too difficult. Mainly, the translation dealt with the creation of lists for each question, in line Latex translations for the math equations, and spitting out the answer key at the end. One important feature of the listener is in the handling of a "no solutions case". It may be the case that due to randomization an equation might have no solutions. There is functionality to handle this in the answer options, so we will allow it. However, the roots function from Symja would return an error to the output, and in order to not have to go back and delete the error message, the solveEquation() function is equipped with the ability to suppress error messages. I just looked up a way to create a new error print stream in java, and found a simple solution. It's a try, finally block, where we first make the error message print nothing (essentially ignoring it), and then returning to normal error handling after each call to solveEquation() to handle any other errors. Basically, any time we call solveEquation(), it ignores and doesn't print math related error for no solutions so we can have interesting cases where "no solutions" may be an answer.
